Tutorial-0: The Web

If we are going to develop interactive applications for the web, we need to understand a little bit about how the web works.  This tutorial is going to give a very basic overview of how the web works, which will allow us to get started with web development, and I will have links you can follow to deepen your understanding.  

This tutorial draws heavily from Chapter 3 of Scott Murray’s “Interactive Data Visualization for the Web”, which is available for free to UC students through the O'Reilly online portal.  

What is the Web? 

When we think of the web, we think of our experience with the web, navigating through a bunch of interlinked pages.  But, if we look underneath this experience, what we actually have is a conversation between web servers and web clients (or browsers).

Let's imagine how this conversation might go.  

The user clicks a link on a web page.  

CLIENT:  "I would like to see what is available at somewebsite.com.  Let me call and see if I can get the latest info"

SERVER: "Hello web client.  I am the server hosting somewebsite.com.  What page are you looking for?" 

CLIENT:  "I would like to retrieve a page at somewebsite.com/news/.

SERVER: "Yes, let me get that for you, one moment."

Code is transmitted from SERVER to CLIENT

CLIENT: "Got it, thanks!"

SERVER: "You are welcome.  Sorry I can't stay and chat, but I have other folks on the line i need to help. Let me know if you need anything else!"

Clients contact servers with requests and servers respond with data.  Servers are connected to the internet, and they run software that is designed to serve web documents when requested.  The servers you think about serve content publicly, and are always on.  You can also run a local server, which means they run on the same computer that you are working on.  Local refers to the fact that only your computer can access the content.  Here's my tutorial on how to run a local server using python simple server.  It is how I often develop web content initially.  Often web servers are written in Apache, and are complex.  You can also write servers using node.js, which is much easier to get started with.  

How do web browsers work? 

Web browsers are software applications take act as the client in the above scenario. They request pages from servers, they receive the code sent by the page, and run it, presenting the results to the user.  The browser interprets the html, javascript and other content sent by the server, and presents it. 

Components: (image credit: https://medium.com/@monica1109/how-does-web-browsers-work-c95ad628a509)

The part of the browser we see, is its user interface.  The tabs, the search bar, the refresh button.  This is the interface which connects the users to the functionality of the browser- retrieving and presenting and enabling us to interact with content on servers.  Between the UI and the backend components is the browser engine, which connects the two.  

The real work of deciding what to present to users based on the code fetched from the servers is done by a rendering engine that interprets the html, the images and other content, and decides how to lay things out. The rendering engine will traverse the html 'content tree' or DOM (which we discuss in the next section).  It parses the css files and style elements.  Parsing involves taking the rules written by developers in html, tokenizing it, grabbing the tags, producing something the rendering engine can read and work off of.  You can read more about it here.    The parser constructs a render tree, which consists in the visual elements in the order they will be displayed.  After constructing the render tree, the rendering engine in the browser will compute the layout of elements.  Elements are specified in hierarchical groups, with different indicated layout rules, such as whether elements are center aligned, or sized to fill the parent container horizontally, etc.  The layout process translates this information into actual positions and sizes for elements on the screen.  Then the rendering engine paints the content.  This means it has a backend UI layer which decides, based on the specified elements, style rules and layouts how to fill in the pixels within the browser window.  Lots of optimization goes on to make sure users have a smooth experience, and content is presented and updated quickly for the user.  There is an open source rendering engine out there if you are curious, called webkit.

Browsers have a javascript interpreter.  The interpreter can fetch external javascript files specified in the code, as well as internal ones.  It interprets the javascript code written by the developer.  Here you can draw on your experience with other languages.  Javascript is not compiled into machine code and run on the machine.  Javascript is interpreted, which means its execution begins immediately.  The browser's javascript interpreter is a bit like a virtual machine, in the sense that you, the developer, do not need to worry about compiling your code for different machines, because it runs within the browser, which is itself compiled into machine code. 

Javascript has a garbage collector.  But, if you create lots of large global variables, these stick around for the entire run of the program, and you could potentially be wasting memory.  This has never been an issue for me, but it is worth keeping in mind.  

The browser runs an event loop, which listens for user actions continuously.  It catches these events (mouse clicks, keyboard key presses....) so that these can be responded to.   Developers can create different event listeners which pay attention for events within the event loop.  You can also set intervals and timeouts, to control the execution of a process over time, like an animation.  

The code you write in Javascript executes within a single thread- it has one call stack, where the operations are pushed.  But then how do we avoid having our code stall while waiting for something to be retrieved or events to run?  This is where we need to think about doing certain tasks asynchronously or taking advantage of the event loop.  

We will get into these topics more as we create web applications, but hopefully this was a helpful general picture of what happens when you write code for the web.
